## Часть 1 EDA

Эти признаки используются как слой конструирования признаков для классической базовой модели и как аналитические признаки для EDA

Артефакты EDA:

## Тематическое моделирование

### Выбор алгоритма

Для тематического моделирования выбран NMF поверх TF-IDF

Обоснование выбора:

- NMF хорошо работает с разреженными TF-IDF матрицами и дает интерпретируемые неотрицательные темы
- Для отзывов и коротких пользовательских текстов NMF обычно дает более понятные top words, чем LDA на счетчиках
- LDA требует более строгой вероятностной предпосылки о генерации документов и чаще нуждается в тщательной лемматизации
- BERTopic мог бы дать более сильные семантические темы, но он тяжелее, требует эмбеддингов и усложняет воспроизводимость

Параметры:

- `topic_count`: `8`
- `topic_top_words`: `12`
- максимальное число TF-IDF признаков: `30000`
- sample для topic modeling: `30000` текстов
- n-grams: `1..2`
- фильтрация признаков: `min_df=5`, `max_df=0.85`

### Варианты улучшения тематического моделирования

- Лемматизация русского текста через `pymorphy3` или `natasha`, чтобы объединить формы слов вроде `врач`, `врача`, `врачи`
- Более качественный список stop words с доменными словами, которые часто встречаются, но плохо различают темы
- Подбор числа тем через coherence score и ручную интерпретацию нескольких значений `k`
- Отдельное моделирование тем внутри каждого sentiment-класса, чтобы не смешивать домен и тональность
- BERTopic или Top2Vec на эмбеддингов предложений для выделения семантических тем, а не только лексических n-грамм
- Удаление дублей и слишком коротких текстов перед topic modeling

## Часть 2 Обучение классификатора

### Разбиение данных

Для transformer-модели используется:

- train: `170901`
- validation: `21098`
- test: `18990`

Test-часть выделяется из `data/raw/train.csv` стратифицированно с `test_size=0.1`, validation берется из `data/raw/valid.csv`

Для базовой модели с признаками использована стратифицированная подвыборка `30000` текстов:

- train: `24000`
- validation: `3000`
- test: `3000`

### Работа с признаками и отбор признаков

Реализован отдельный классическая базовая модель:

- очистка текста: приведение к нижнему регистру, замена `ё` на `е`, нормализация URL, чисел и пробелов
- TF-IDF признаки: униграммы и биграммы
- ручные признаки: длина, число слов, средняя длина слова, число знаков пунктуации, число цифр, доля заглавных символов
- отбор признаков: `SelectKBest(chi2)` по TF-IDF признакам
- классификатор: `LogisticRegression` с `class_weight="balanced"`

Обоснование:

- TF-IDF + n-граммы дает сильный и интерпретируемую базовую модель для тональности
- chi2 подходит для отбора текстовых признаков в задачах классификации с неотрицательными TF-IDF значениями
- ручные признаки добавляют информацию о стиле текста, которую n-граммы могут не отражать напрямую
- Logistic Regression быстрее transformer-модели и помогает проверить вклад фичей без тяжелого обучения

Ключевые признаки, отобранные через chi2:

`благодарность`, `спасибо`, `отель`, `врач`, `огромное`, `врача`, `дозвониться`, `прием`, `поблагодарить`, `номера`, `хочу`, `завтрак`, `хочу поблагодарить`, `огромное спасибо`, `сказали`, `трубку`, `сказала`, `профессионализм`, `выразить`, `метро`

Основная метрика базовой модели с признаками:

| Метрика | Значение |
|---|---:|
| Macro F1 | 0.6800 |


### Выбор модели классификации

Основная модель: `cointegrated/rubert-tiny2`

Обоснование выбора:

- Модель предобучена на русском языке и лучше учитывает контекст, чем TF-IDF базовая модель
- `rubert-tiny2` компактнее больших RuBERT моделей, поэтому подходит для сравнения CPU/GPU скорости
- Transformer-модель лучше переносит смысловые зависимости и порядок слов
- Альтернативы вроде Logistic Regression быстрее, но показывают заметно более низкий macro F1 на базовой модели

### Параметры обучения

Параметры вынесены в `configs/config.yaml`

| Параметр | Значение |
|---|---:|
| max_len | 256 |
| batch_size | 32 |
| learning_rate | 2e-5 |
| epochs | 2 |
| warmup_ratio | 0.1 |
| weight_decay | 0.01 |
| seed | 42 |

Обоснование параметров:

- `max_len=256` ограничивает стоимость инференса и сохраняет достаточно контекста для отзывов
- `batch_size=32` балансирует скорость и потребление памяти
- `learning_rate=2e-5` является типичным безопасным значением для fine-tuning BERT-подобных моделей
- `epochs=2` снижает риск переобучения и сохраняет разумное время обучения
- `warmup_ratio=0.1` стабилизирует первые шаги fine-tuning
- `class_weight` используется из-за дисбаланса классов

### Метрика качества и функция потерь

Основная метрика: macro F1

Обоснование:

- Классы несбалансированы, поэтому главная метрика должна одинаково учитывать качество по каждому классу
- Macro F1 одинаково учитывает каждый класс и лучше соответствует задаче трехклассовой тональности

Функция потерь: взвешенная кросс-энтропия

Обоснование:

- Cross entropy стандартна для многоклассовой классификации
- Веса классов компенсируют дисбаланс и повышают вклад меньших классов

### Варианты улучшения качества и скорости классификатора

- Удалить дубли и проверить разбиение на утечки одинаковых текстов между split
- Подобрать `max_len`, `learning_rate`, `epochs`, `weight_decay` и scheduler через validation
- Добавить domain-aware split или отдельную валидацию по доменам, чтобы проверить переносимость
- Использовать lemmatization и словарные признаки в ансамбле с transformer-моделью
- Попробовать более сильную русскую модель, например rubert-base, если качество важнее скорости
- Сделать knowledge distillation из более крупной модели в компактную
- Использовать ONNX Runtime с корректной динамической или статической квантизацией
- Сократить `max_len` до 128, если потеря качества приемлема, так как CPU скорость почти удваивается
- Использовать batching и кеширование tokenizer outputs для массового инференса

## Часть 3 Оптимизация классификатора

### Выбор способа оптимизации

Выбрана динамическая INT8-квантизация для Linear-слоев PyTorch модели

Обоснование:

- Метод не требует отдельного калибровочного датасета
- Подходит для CPU-инференса и легко воспроизводится
- Для BERT-подобных моделей Linear-слои занимают существенную часть вычислений
- Альтернативы ONNX-квантизация и pruning сложнее в отладке и могут требовать больше времени на проверку качества

### Результаты оптимизации

Качество исходной модели и dynamic INT8 дополнительно проверено на одинаковой подвыборке `2000` test-текстов

Основная метрика оптимизации:

| Модель | Macro F1 |
|---|---:|
| Исходная PyTorch CPU | 0.7525 |
| Dynamic INT8 CPU | 0.7548 |


Скорость из бенчмарка:

| Модель | Устройство | Текстов в секунду | мс на текст |
|---|---|---:|---:|
| Исходная PyTorch | CPU | 809.27 | 1.2357 |
| Dynamic INT8 | CPU | 605.84 | 1.6506 |

Вывод по оптимизации:

- Dynamic INT8 почти не ухудшила качество на проверочной подвыборке
- На текущем окружении dynamic INT8 оказалась медленнее исходная модель CPU, поэтому для сдачи результат важен как проверенная оптимизационная попытка, но не как финальный самый быстрый вариант
- Самый быстрый проверенный вариант без потери инфраструктурной простоты: исходная модель PyTorch CPU с `max_len=128`, `1783.62` текстов/сек
